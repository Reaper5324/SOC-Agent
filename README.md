# SOC Agent

This project is a small Security Operations Center (SOC) analysis pipeline written in Python. Its job is to read network telemetry from Zeek and Suricata, convert that raw data into a common internal event format, apply simple detection logic, group suspicious activity into incidents, optionally send those incidents to Ollama for a language-model-based summary, and finally write a human-readable report. The design is intentionally modular, which makes it a good example for a third-year computer science student because each file has a focused responsibility and participates in a clear dataflow from raw logs to analyst-facing output.

The overall system follows a pipeline architecture. Data starts in the `Data` folder as log files, flows through parser modules that normalize the logs into Python dictionaries, moves into a rule engine that selects suspicious events, then enters the analyst layer which turns events into incident summaries, and finally ends in `output/report.txt`. The recent logging addition also lets the user watch this pipeline stage by stage on the terminal, which improves observability without changing the core analysis logic.

## System Dataflow

When `main.py` runs, it first loads environment-based configuration through `config.py`, then starts terminal logging through `utils/logger.py`. After that it calls `parse_conn_log()` from `parsers/zeek_parser.py` and `parse_eve()` from `parsers/suricata_parser.py`. These two functions read different file formats but produce similarly shaped dictionaries so the rest of the system does not need to care where an event came from. The combined event list is then passed into `apply_rules()` in `Detection/rules.py`, which filters or enriches records that look suspicious. The suspicious events are passed into `analyse()` in `agent/analyst.py`, where events are grouped into incidents using source IP, destination IP, and event type. If Ollama is enabled, `analyse()` calls `generate_incident_summary()` in `agent/ollama_client.py` to ask a local language model for a better incident write-up. Once the findings are ready, `main.py` writes them into `output/report.txt`.

This dataflow shows an important software engineering choice: the project separates parsing, detection, analysis, and output into different layers. That separation reduces complexity in each file, makes testing easier, and limits the amount of code that needs to change if one part of the pipeline changes. For example, a new parser could be added without rewriting the rule engine, as long as it produces the same event structure.

## Use Case

A realistic use case is a junior SOC analyst or security student who has collected Zeek connection logs and Suricata alert logs from a small lab environment. Instead of manually reading large text files line by line, the user runs this project to automatically highlight suspicious network behavior such as Suricata alerts, unusual UDP traffic, or repeated connections from one source IP within a short time window. The program then groups related events into incidents and produces a report that explains why the activity matters and what actions should be taken next. If Ollama is running locally, the report can include richer natural-language analysis generated by the selected model such as `tinyllama`.

## File-by-File Explanation

### `main.py`

`main.py` is the orchestration layer and acts as the program entry point. Its purpose is to coordinate the whole SOC workflow from start to finish without containing too much detailed logic itself. This is a strong design choice because entry-point files should usually be easy to read and should describe the high-level steps of the system. In this file, the `main()` function loads configuration, logs progress messages, parses the Zeek and Suricata datasets, merges the resulting events, applies detection rules, sends suspicious events for incident analysis, and writes the final report. The `write_report()` function is responsible for formatting findings into a plain-text report. It creates the output directory if needed, builds the report line by line, and writes the result to disk.

The methods in `main.py` are simple but purposeful. `main()` uses composition rather than implementing parsing or detection itself. This improves cohesion because the file focuses on orchestration only, and it keeps coupling at a moderate level because `main.py` depends on several modules but mostly through clear function calls. `write_report()` transforms structured finding dictionaries into human-readable text. It checks whether findings exist, loops through incidents, and prints fields such as incident name, severity, analysis source, event count, escalation decision, explanation, and next actions. The dataflow role of `main.py` is therefore to receive data from almost every other core file and move it forward to the next stage. It is highly central to the system, which means it has moderate-to-high coupling, but its internal cohesion is still high because everything it does supports one responsibility: running the pipeline.

### `config.py`

`config.py` is responsible for configuration management. Its purpose is to make environment-specific values such as the Ollama host, model name, timeout, and enable/disable flag available to the rest of the application in a clean way. This is important because hard-coding these values throughout the project would make the system harder to deploy, test, and maintain. The file defines `BASE_DIR` and `ENV_FILE` so it knows where to look for a local `.env` file. The `load_env_file()` function opens that file if it exists, reads it line by line, ignores blank lines and comments, splits each valid line into a key and value pair, and inserts those values into `os.environ` only if they are not already set. This design respects externally defined environment variables and avoids unexpectedly overwriting them.

The other two methods are `get_bool()` and `get_config()`. `get_bool()` converts user-friendly strings such as `true`, `yes`, `on`, or `1` into a real Python boolean. `get_config()` collects the final settings into a single dictionary that the rest of the system can use. This file has high cohesion because every method contributes directly to configuration loading or normalization. Its coupling is low to moderate because many other files depend on it, but it itself depends only on Python standard library modules. In the dataflow, `config.py` sits near the beginning. It does not process security data directly, but it controls whether the later analysis stage will use Ollama and which model it will call.

### `.env.example`

`.env.example` is not executable code, but it is still an important project file because it documents the expected runtime configuration. Its purpose is to show the user which environment variables need to be defined and what kind of values the system expects. In this case, it includes `OLLAMA_ENABLED`, `OLLAMA_HOST`, `OLLAMA_MODEL`, and `OLLAMA_TIMEOUT`. The design choice here is a common one in modern software projects: keep configuration outside source code so the same codebase can run in different environments without modification. For a student, this file is best understood as a template rather than a program component.

In terms of interaction, `.env.example` works indirectly with `config.py`. A user copies it to `.env`, edits the values, and then `config.py` loads them at runtime. Its cohesion is high because all of its contents serve one purpose: documenting configuration. Its coupling is low because it is not imported by the Python modules, but it still has an operational relationship with the rest of the system.

### `parsers/zeek_parser.py`

`parsers/zeek_parser.py` is responsible for transforming Zeek connection logs into the project’s shared event structure. Its main method, `parse_conn_log(file_path)`, opens the Zeek `conn.log` file, ignores metadata lines that start with `#`, splits each remaining line into fields, checks that enough columns exist, and then builds a Python dictionary containing normalized keys such as timestamp, source IP, destination IP, ports, protocol, service, event type, description, severity, severity score, and source tool. The method returns a list of these dictionaries.

The key design choice in this file is normalization. Zeek logs have their own raw format, but the rest of the system should not need to understand that format. By converting the data early into a common schema, later modules such as the rule engine and analyst layer can treat all events consistently. This parser has high cohesion because it only parses Zeek data. Its coupling is low because it exposes one clear function and does not need to know about the rest of the analysis pipeline. In the system dataflow, this file is one of the first transformation stages. It sends its output directly to `main.py`, which merges it with parsed Suricata events before rule evaluation.

### `parsers/suricata_parser.py`

`parsers/suricata_parser.py` performs the same architectural role for Suricata that `zeek_parser.py` performs for Zeek, but it works with JSON-based `eve.json` data. Its purpose is to read Suricata events and keep only the records that are relevant to this project’s current scope, which are alert events. The method `parse_eve(file_path)` opens the file, strips blank lines, attempts to parse each line as JSON, skips malformed JSON safely, ignores records whose `event_type` is not `alert`, and then constructs a normalized event dictionary. It extracts alert-specific fields such as signature text, signature ID, and alert severity while also mapping network details such as source IP, destination IP, ports, and protocol.

This parser also follows the normalization design choice, which is good because it means `Detection/rules.py` and `agent/analyst.py` do not need parser-specific logic. The file has high cohesion because all of its logic is dedicated to Suricata alert extraction and normalization. Its coupling is low because it provides one service to the rest of the program through a simple function interface. In the dataflow, it runs in parallel conceptually with the Zeek parser: both create lists of normalized events, and those lists are then merged in `main.py`.

### `Detection/rules.py`

`Detection/rules.py` implements the first actual detection layer of the system. Its purpose is to decide which normalized events should be treated as suspicious and, when needed, enrich those events with more meaningful severity or descriptions. The main method is `apply_rules(events)`. It loops through all normalized events and applies several rules. First, every Suricata alert is automatically considered suspicious. Second, any Zeek connection that uses UDP is marked as suspicious and enriched with a description and medium severity. Third, the file looks for a simple frequency-based pattern: if the same source IP appears in five or more connection events within a ten-minute window, it creates a higher-severity event labeled as possible brute force or scanning activity.

The helper methods `_append_unique()` and `_to_epoch_seconds()` support `apply_rules()`. `_append_unique()` prevents duplicate suspicious events from being added by creating a tuple-based event key and tracking previously seen records. `_to_epoch_seconds()` converts timestamps to floating-point seconds when possible, which makes the time-window rule easier to compute. These helper methods are good design choices because they keep the main rule loop from becoming too cluttered. This file has high cohesion because every function is related to rule-based detection. Its coupling is moderate because it depends on the event schema produced by both parser files, especially fields such as `event_type`, `source_tool`, `timestamp`, `src_ip`, and `protocol`. In the dataflow, it acts as the filtering and enrichment stage between raw event parsing and higher-level incident analysis.

### `agent/analyst.py`

`agent/analyst.py` is the incident analysis layer. Its purpose is to take suspicious events and turn them into structured incident findings that are more useful to a human analyst. The main method is `analyse(events)`. It first groups events into incidents using a tuple of `(src_ip, dest_ip, event_type)` as the grouping key. This is a simple but effective design because it groups together activities that are likely related while staying easy to understand and implement. After grouping, the file computes a severity score, builds a fallback incident summary, and then optionally improves that summary by sending the grouped event data to Ollama.

Several helper methods support this process. `normalise_severity(event)` converts different severity representations into a numeric score. `severity_label(score)` converts numeric scores back into readable labels such as `high` or `medium`. `_recommended_actions(event, highest_score)` generates action suggestions based on source tool, protocol, and severity. `_build_fallback_finding(...)` creates a complete incident dictionary even if no language model is available. `_analyse_with_ollama(...)` builds a prompt, sends it to the Ollama client, validates the returned structure, and merges the language-model response with fallback defaults. `_build_incident_prompt(...)` constructs the exact prompt sent to the model, including schema instructions and the serialized event context. `_severity_score_from_label(...)` maps a severity label back into a numeric ranking. The file also logs progress for each incident so the user can see which analysis stage is running.

This module has high cohesion because all of its functions contribute directly to incident-level reasoning. Its coupling is moderate. It depends on `config.py` for runtime options, on `agent/ollama_client.py` for model communication, and on the event schema created earlier in the pipeline. However, those dependencies are well-structured and purposeful rather than chaotic. In the dataflow, this module is where suspicious events stop being just records and start becoming analyst-facing findings.

### `agent/ollama_client.py`

`agent/ollama_client.py` is responsible for communication with the local Ollama service. Its purpose is to isolate HTTP request logic from the rest of the system so that the analyst module does not need to know implementation details about URLs, JSON encoding, request headers, or response parsing. The single public method, `generate_incident_summary(host, model, prompt, timeout=60)`, constructs the Ollama endpoint URL, builds a JSON payload containing the model name, prompt, non-streaming mode, and JSON output format, then sends that payload using Python’s `urllib` library. It catches connection-related errors and raises a readable `RuntimeError` so the rest of the application can fall back gracefully if Ollama is unavailable.

After receiving the response, the method parses the JSON body, extracts the `response` field, and then parses that field again because the model is expected to return JSON text. This two-step decode is an important implementation detail because the outer API response and the inner model response are not the same JSON object. This file has very high cohesion because it does one thing only: model communication. Its coupling is low to moderate because only the analyst module needs it directly. In the dataflow, it sits off the main path as an optional branch called by `agent/analyst.py` when richer natural-language analysis is needed.

### `utils/logger.py`

`utils/logger.py` provides shared terminal logging for the project. Its purpose is to make runtime progress visible in a consistent format without scattering logging configuration across multiple files. The method `setup_logging()` configures Python’s logging system with an INFO level, a timestamped format, and standard output as the destination. It also uses a module-level flag to avoid configuring logging more than once, which is a good defensive design choice because repeated logger initialization can create duplicate messages. The method `get_logger(name)` ensures logging is set up and then returns a named logger for the calling module.

This file has high cohesion because every line supports the same concern: observability. Its coupling is low because it depends only on the standard library and exposes a very small interface to the rest of the system. In terms of interaction, `main.py` and `agent/analyst.py` currently use it to report pipeline progress and Ollama activity. In the dataflow, it does not transform security data directly, but it makes the dataflow visible to the user while the program executes.

### `Data/zeek/conn.log`

`Data/zeek/conn.log` is a sample Zeek input file. Its purpose is to provide raw network connection records for the parser to process. It is not Python code, but it is part of the system because it supplies the data consumed by `parsers/zeek_parser.py`. Its interaction with the rest of the project is therefore through file I/O rather than imports. Its cohesion is naturally high because it contains one kind of telemetry, and its coupling is low because only the Zeek parser needs to understand its exact structure.

### `Data/suricata/eve.json`

`Data/suricata/eve.json` is a sample Suricata log file in line-delimited JSON format. Its purpose is to provide alert telemetry that can be parsed into suspicious events. `parsers/suricata_parser.py` reads it line by line and selects only the alert records relevant to the current system design. Like the Zeek dataset, it has high cohesion because it is dedicated to one source of telemetry and low coupling because only one parser needs to know its raw structure.

### `Data/suricata/Organizer.ps1`

`Data/suricata/Organizer.ps1` is a PowerShell script that appears to be unrelated to the core SOC analysis pipeline. Its purpose seems to be file organization on a Windows system rather than security log analysis. It loops through files in a target directory, chooses a destination folder based on file extension, moves files, and tries to log those moves. From a documentation perspective, it is useful to mention that this script is currently separate from the Python SOC workflow and is not called anywhere by `main.py` or the analysis modules. It also contains several spelling and syntax issues such as `Move_Item`, `$destination - Join-Path`, and misspelled variable names, which suggests it may be an experimental or practice script rather than a production component.

Its cohesion is moderate because all of its logic relates to file organization, but that responsibility does not align with the main project domain. Its coupling with the SOC system is effectively zero because no Python file imports or executes it. If this repository is being submitted academically, it would be wise to clarify in class documentation that this script is auxiliary and not part of the incident-analysis pipeline.

### `output/report.txt`

`output/report.txt` is the generated output artifact of the system. Its purpose is to present final incident findings in a form that a human can read without inspecting Python objects or raw log data. It is written by `write_report()` in `main.py`, and its contents reflect the results produced by the full pipeline. The example currently shows fallback analysis because Ollama was unreachable during the sample run, but the same file would contain model-enriched findings if Ollama were available. Its cohesion is high because it stores one kind of output, and its coupling is low because it is only written by one part of the system and consumed mainly by the user.

## Interactions Between Files

The interaction pattern in this project is mostly hierarchical rather than circular, which is a healthy design choice. `main.py` sits at the top and coordinates the run. `config.py` and `utils/logger.py` provide shared services used by other files. The parser files provide normalized events to `main.py`. `Detection/rules.py` receives those normalized events and sends suspicious ones onward. `agent/analyst.py` converts suspicious events into findings and optionally relies on `agent/ollama_client.py` to improve those findings. Finally, `main.py` writes the report to `output/report.txt`. This means dependencies mostly point forward through the pipeline rather than back and forth between modules, which reduces the risk of tangled code.

Another good feature is that the system uses a shared event dictionary format as an interface between stages. This choice lowers coupling because modules communicate through data structures rather than deep knowledge of each other’s internal implementation. At the same time, there is still some semantic coupling because later files expect keys such as `src_ip`, `dest_ip`, `event_type`, `severity`, and `source_tool` to exist. That is a normal tradeoff in lightweight Python systems, although in a larger project this might be improved further with dataclasses or typed models.

## Coupling and Cohesion Summary

Overall, the project shows generally good cohesion. Each core file has a narrow and understandable role: parsing, detection, analysis, model communication, configuration, logging, or orchestration. This is a strength because students and future maintainers can reason about one concern at a time. The strongest examples of high cohesion are `agent/ollama_client.py`, `utils/logger.py`, and the two parser files, because each of these modules focuses on one service only.

The coupling level across the project is mostly low to moderate. The highest-coupled file is `main.py` because it interacts with almost every stage of the pipeline, but that is expected for an entry-point orchestrator. `agent/analyst.py` also has moderate coupling because it combines configuration access, grouping logic, fallback analysis, and optional Ollama use. The rest of the modules are more loosely coupled and are connected through straightforward function calls. From a software engineering perspective, this is a solid structure for a small to medium academic project because it balances simplicity with modularity.

## Design Choices

One major design choice is the use of dictionaries as the common event representation. This keeps the code simple and flexible, which is helpful in student projects and prototypes. The disadvantage is that dictionaries do not enforce structure as strictly as classes or typed models, so a missing key could cause runtime issues if the project grows. Another design choice is the fallback mechanism in `agent/analyst.py`. Even if Ollama fails, the system still produces a useful report. That improves robustness and is a strong practical decision because external AI services are not always available.

A third design choice is the addition of `utils/logger.py` for observable execution. This is important in security tooling because users often want to know whether the parser is running, how many events were loaded, when rules are being applied, and when incident analysis is taking place. A fourth design choice is the separation of detection from analysis. `Detection/rules.py` decides whether something is suspicious, while `agent/analyst.py` decides how to describe and prioritize it. That distinction reflects real SOC workflows and makes the code easier to evolve.

## Final Evaluation

For a third-year computer science project, this repository demonstrates a clear and teachable architecture. The code is small enough to understand fully, but it still includes meaningful concepts such as normalization, rule-based detection, incident grouping, fallback handling, local AI integration, configuration management, and runtime logging. The strongest architectural qualities are high cohesion in most modules and a mostly linear, easy-to-follow dataflow. The main area where the design could evolve in the future would be stronger typing for events and perhaps more formal separation between domain models and raw dictionaries, but for the current scope the project is well-structured and suitable for explaining core software engineering principles.
